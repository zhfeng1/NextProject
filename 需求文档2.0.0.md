# NextProject 需求文档 v2.0.0（优化版）

## 1. 产品定位

### 1.1 产品愿景
NextProject 是一个企业级 AI 驱动的网站开发平台，旨在通过多 AI 代理协作，实现从需求到部署的全流程自动化，降低 Web 应用开发门槛，提升开发效率 10 倍以上。

### 1.2 目标用户
- **产品经理**：快速验证想法，无需编写代码
- **前端开发者**：自动生成原型，专注核心业务逻辑
- **全栈团队**：并行开发多个项目，统一管理和部署
- **DevOps 工程师**：标准化部署流程，降低运维复杂度

### 1.3 核心价值
- **极速原型**：5 分钟从想法到可运行网站
- **多站点管理**：单一平台管理数百个子网站
- **AI 协作**：Codex + Claude + Gemini 三重保障
- **一键部署**：本地预览 → Apollo 生产环境无缝切换

### 2.2 服务组件升级

#### 主服务 (main-service)
**当前**
- FastAPI + SQLite
- 单容器运行

**优化**
- 拆分为微服务：
  - `api-gateway`：统一入口、认证、限流
  - `site-manager`：站点生命周期管理
  - `task-scheduler`：任务调度和监控
  - `ai-proxy`：AI Provider 统一接口
  - `deploy-engine`：部署流程编排

#### 子网站系统
**当前**
- 进程级隔离
- 端口硬编码

**优化**
- **Docker 容器隔离**：每个子网站独立容器
- **动态端口分配**：Traefik 反向代理 + 域名路由
- **资源限制**：CPU/内存配额（Docker limits）
- **快速扩缩容**：容器编排（Docker Swarm / K8s）

#### 数据库升级
**当前**
- SQLite（单文件）

**优化**
- **PostgreSQL**：支持并发、事务、复杂查询
- **Redis**：
  - 任务队列持久化（Celery + Redis）
  - 会话缓存
  - 实时任务状态（WebSocket 推送）
- **MinIO**：
  - 网站代码归档
  - 测试产物存储
  - 部署镜像备份

### 3.3 AI 代理增强

#### 智能路由
**当前**：用户手动选择 Provider

**优化**：
- 任务分类模型：
  - UI 调整 → Codex（擅长前端）
  - API 开发 → Claude（擅长后端逻辑）
  - 性能优化 → Gemini（擅长代码优化）
- 失败自动重试：Provider A 失败 → 自动切换 Provider B

#### 上下文增强
**当前**：单次 prompt

**优化**：
- 对话式开发：
  - 用户："添加用户登录功能"
  - AI："需要 OAuth 还是邮箱登录？"
  - 用户："邮箱登录"
  - AI：生成完整代码 + 数据库迁移脚本
- 知识库集成：
  - 自动注入站点现有代码结构
  - 参考历史成功案例
  - 遵循企业编码规范

#### 协作模式
**新增功能**
- Multi-Agent 并行：
  - Agent 1：生成前端组件
  - Agent 2：同时生成后端 API
  - Agent 3：编写单元测试
- Code Review：
  - 另一个 AI 审查生成代码
  - 自动修复 ESLint / Pylint 错误

### 3.4 测试系统升级

#### 自动化测试套件
**当前**：Playwright 冒烟测试

**优化**：
- **单元测试**：自动生成 pytest / Jest 测试
- **集成测试**：API 端到端测试（Postman Collections）
- **性能测试**：Lighthouse CI（性能评分）
- **安全测试**：OWASP ZAP 扫描

#### 持续测试
- 每次代码变更自动触发测试
- 测试报告可视化（覆盖率、趋势图）
- 失败自动回滚

**API 示例**
```
POST /api/tasks（task_type=test_suite）
GET  /api/sites/{site_id}/test-reports
```

### 3.5 部署流程优化

#### 多环境管理
**当前**：local / apollo

**优化**：
- **环境隔离**：
  - Development（开发）
  - Staging（预发布）
  - Production（生产）
- **环境变量管理**：
  - 每个环境独立配置
  - 密钥加密存储（Vault）
- **灰度发布**：
  - 10% 流量 → Staging
  - 验证通过 → 100% 流量 → Production

#### CI/CD 集成
**新增功能**
- GitHub Actions / GitLab CI 集成
- 自动构建流水线：
  1. 代码变更 → 触发构建
  2. 运行测试套件
  3. 构建 Docker 镜像
  4. 推送到 Harbor
  5. 部署到 K8s
  6. 发送通知（Slack / 钉钉）

#### 部署策略
- **蓝绿部署**：零停机切换
- **金丝雀发布**：逐步扩大新版本流量
- **自动回滚**：健康检查失败 → 回滚到上一版本

### 3.6 监控与告警

#### 实时监控
**新增功能**
- **性能指标**：
  - 响应时间（P50、P95、P99）
  - 吞吐量（QPS）
  - 错误率
- **资源监控**：
  - CPU / 内存使用率
  - 网络流量
  - 磁盘 I/O
- **业务指标**：
  - 活跃站点数量
  - 任务成功率

## 4. 非功能需求（强化版）

### 4.1 性能要求

| 指标 | 当前 | 目标 |
|------|------|------|
| 站点创建速度 | ~3 秒 | <1 秒（容器预热池）|
| 并发任务数 | 2 | 50+（Celery 分布式）|
| API 响应时间（P95）| 未测 | <200ms |
| 同时在线站点 | ~100 | 1000+（K8s 自动扩缩容）|

### 4.2 安全要求

**加密**
- 传输层：TLS 1.3
- 存储层：数据库字段加密（API 密钥、密码）
- 密钥管理：HashiCorp Vault

**访问控制**
- OAuth 2.0 / OIDC
- API 限流：100 req/min/user
- CORS 白名单

**审计日志**
- 所有操作记录（用户、时间、IP、操作）
- 日志不可篡改（写入 S3 + Object Lock）

**漏洞扫描**
- 依赖库扫描（Dependabot）
- 容器镜像扫描（Trivy）
- 定期渗透测试

### 4.3 可靠性

**高可用**
- 服务多副本（3 实例）
- 数据库主从复制 + 自动故障转移
- 跨可用区部署

**容错**
- 服务降级：AI Provider 故障 → 降级为模板生成
- 熔断机制：下游服务故障 → 快速失败
- 重试策略：指数退避 + 抖动

**备份与恢复**
- 数据库每日全量备份 + 每小时增量备份
- 站点代码实时同步到 MinIO
- RTO < 1 小时，RPO < 5 分钟

### 4.4 可扩展性

**水平扩展**
- 无状态服务：直接增加副本
- 有状态服务：Redis Cluster、PostgreSQL 分片

**垂直扩展**
- 资源自动调整（K8s VPA）

**多云支持**
- 抽象部署接口：
  - AWS ECS
  - 阿里云 ACK
  - 腾讯云 TKE
  - 自建 K8s

```sql
CREATE TABLE task_queue (
```

## 6. API 设计优化

### 6.1 RESTful 规范

**当前问题**
- 不一致的命名（/api/build-site vs /api/sites）
- 混合 POST 用途（创建 + 操作）

**优化后**
```
# 资源风格
GET    /api/v2/sites
POST   /api/v2/sites
GET    /api/v2/sites/{id}
PATCH  /api/v2/sites/{id}
DELETE /api/v2/sites/{id}

# 操作风格（子资源）
POST   /api/v2/sites/{id}/start
POST   /api/v2/sites/{id}/stop
POST   /api/v2/sites/{id}/deploy
POST   /api/v2/sites/{id}/rollback

# 批量操作
POST   /api/v2/sites/batch-delete
```

### 6.2 WebSocket 实时推送

**当前**：轮询日志（效率低）

**优化**：
```javascript
// 客户端
const ws = new WebSocket('wss://api.example.com/ws/tasks/123/logs');
ws.onmessage = (event) => {
  const log = JSON.parse(event.data);
  console.log(log.line);
};

// 服务端（FastAPI）
@app.websocket("/ws/tasks/{task_id}/logs")
async def task_logs_ws(websocket: WebSocket, task_id: str):
    await websocket.accept()
    async for log in subscribe_task_logs(task_id):
        await websocket.send_json(log)
```

### 6.3 GraphQL（可选）

**适用场景**
- 前端需要灵活查询（避免多次 REST 调用）
- 站点详情页：一次查询获取站点 + 任务 + 版本

**示例**

## 9. 成本优化

### 9.1 AI 调用成本

**当前风险**
- 无限制调用可能导致高额费用

**优化策略**
- 用户配额：
  - 免费版：100 次 AI 调用/月
  - Pro 版：1000 次/月
  - Enterprise：无限制
- Token 预估：调用前预估成本，超出阈值提示用户
- 缓存复用：相似 prompt 返回缓存结果

### 9.2 资源成本

**容器预热池**
- 预先启动 10 个空闲容器
- 用户创建站点时立即分配（秒级响应）
- 回收空闲超过 1 小时的容器

**按需计费**
- 站点休眠：24 小时无访问 → 停止容器
- 唤醒策略：首次访问时自动启动（冷启动 ~5 秒）

## 11. 问题与建议

### 关键问题

1. **安全隐患**：API 密钥明文存储
   - **建议**：集成 HashiCorp Vault

2. **单点故障**：SQLite 无法水平扩展
   - **建议**：迁移到 PostgreSQL + 主从复制

3. **资源隔离**：子网站进程级隔离不足
   - **建议**：改为 Docker 容器 / K8s Pod

4. **成本控制**：无 AI 调用额度限制
   - **建议**：实现用户配额系统

5. **监控盲区**：无系统级监控
   - **建议**：部署 Prometheus + Grafana
